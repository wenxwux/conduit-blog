「どのモデルが最良か?」は間違った問いです。正しい問いは「*このタスクに*いちばんコスパが良いのはどれか」。なぜなら、もっとも高性能なモデルは私たちが実際に頼む仕事の半分にはオーバースペックで、もっとも安いモデルは残り半分でひそかに失敗するからです。そこでリーダーボードを信じる代わりに、同じタスク群を 5 つのモデルで走らせ、品質・速度・コストをまとめて見てみました。その枠組みと分かったことを紹介します。

## セットアップ

ベンチマークの小手先ではなく、実際の仕事を反映した固定のタスク群を使いました。

- **コード:**仕様から小さな機能を実装し、さらにスタックトレースを与えてバグを修正。
- **リファクタ:**散らかった 200 行のファイルを整理。
- **推論:**検証可能な答えのある多段階のロジック問題。
- **文章:**密度の高い段落を分かりやすく書き直す。
- **抽出:**非構造のテキストから構造化 JSON を取り出す。

各タスクを、フラッグシップ・中堅・小型/安価の各層にまたがる 5 つのモデルで走らせました。出力品質を手作業で 1〜5 点で採点し、応答を計測し、token コストを記録しました。あえてモデルごとの正確な数値は公開しません——それらは絶えず変動し、あなたのプロンプトにも依存するからです。しかし*パターン*は安定しており、役に立つのはそこです。

## 「価値(value)」とは実際に何か

指標は生の品質ではなく、1 ドルあたりの価値です。

```
value = quality_score / cost_for_the_task
```

5/5 を取るフラッグシップが、4.5/5 を取る中堅モデルの 8 倍のコストなら、ほとんどのタスクにとって*コスパは悪い*。最後の 0.5 点に大きな割増を払っているわけで、それが意味を持つのは、その 0.5 点が「出荷できるか否か」を分けるときだけです。

## 持ちこたえたパターン

**1. 簡単で構造化されたタスクでは各層が収束する。**
抽出と分類は 5 つのモデルでほぼ同一に見えました——もっとも安いモデルでさえ、きれいで正しい JSON を生成しました。ここでフラッグシップに払うのは純粋な無駄。コスパの勝者は小型モデルが圧倒的です。

**2. 難しい推論では差は本物。**
多段階のロジック問題とより厄介なバグ修正で差がつきました。フラッグシップと強力な中堅モデルは正解し、安価なモデルは自信満々に間違った答えを出しました。割増が正当化されるのはまさにここ——*正しそうに見える*間違った答えは、もっとも高くつく出力です。

**3. 中堅こそ静かなコスパ王。**
下書き、リファクタ、標準的なコーディングを通じて、良い中堅モデルはフラッグシップ品質の約 90% を、ほんの一部の価格で達成しました。日々の仕事の大半にとって、ここがスイートスポット。私はここをデフォルトにし、本当に難しいものに当たったときだけ格上げします。

**4. 速度は思った以上に重要。**
対話的なコーディングでは、品質がわずかに低くても体感で速いモデルのほうが、*作業しやすい*と感じることが多々ありました——フィードバックループが締まり、1 分あたりの反復が増えるからです。人間がループにいるなら、レイテンシも価値の一部です。

## 私が実際に使っている判断表

| タスク | コスパ最良の選択 | 理由 |
|---|---|---|
| 分類 / 抽出 / ルーティング | 小型・安価 | 品質が収束、コストが支配的 |
| 下書き、要約、標準コード | 中堅 | 品質約 90%、コスト大幅減 |
| 既知ファイルのリファクタ | 中堅 | 機械的なので割増を節約 |
| 難しいバグ / あいまいな設計 | フラッグシップ | ここでは 0.5 点に価値がある |
| 対話的ペアコーディング | 速い中堅 | ループ速度が限界品質に勝る |

## 自分でテストを走らせる方法(私のを信じないで)

モデルは毎月変わります。意味のある比較は*あなたの*プロンプトでのものだけです。こうしましょう。

1. 実際のワークロードを代表する 5〜10 個のタスクを選ぶ。
2. プロンプトを一度書き、候補モデルに変えずに走らせる。
3. 品質をブラインドで採点する(どのモデルの出力か隠す)。
4. タスクごとにコストとレイテンシを記録する。
5. `quality / cost` を計算し、全体ではなくタスク種別ごとに見る。

実務上のネックはたいてい配管です——プロバイダごとに SDK・キー・エンドポイントが違うと A/B テストが面倒になります。私は 5 つすべてを 1 つのゲートウェイ、1 つのエンドポイント経由で走らせるので、モデルの切り替えはリクエスト内の文字列を変えるだけ。これで「どのモデルがコスパ最良か」が研究プロジェクトから、新モデルが出るたびに再実行できる 5 分の実験になります。

## まとめ

唯一の最良モデルは存在せず、あるのはタスクごとの最良の*フィット*です。高くつく間違いは、間違ったフラッグシップを選ぶことではなく、簡単で大量の仕事をフラッグシップに送ること自体です。デフォルトは強力な中堅モデルにし、構造化タスクには安価へ落とし、割増に値する難しい推論にだけフラッグシップへ格上げする。1 ドルあたりの価値を自分のタスクで測り、モデルが変わったら測り直す——必ず変わるのですから。
