Claude Code から得たいちばん大きな高速化は、速いモデルではありませんでした——すべてを 1 つの会話でやるのをやめることを学んだことです。サブエージェントを使えば、自己完結した作業のかたまりを、それぞれ独自のコンテキストウィンドウで動く別のエージェントに引き渡し、結果を報告させられます。うまくやれば、「長い 1 本のスレッドに付きっきり」から「5 つのタスクを起動して結果をレビューする」に変わります。

私が実際にどう使っているかを紹介します。

## サブエージェントとは実際に何か

サブエージェントは、メインエージェントが起動する新しい Claude Code インスタンスです。独自のコンテキストウィンドウを持ち、1 つの仕事をして、最終サマリーを返します——途中で読んだ生のファイルの羅列ではありません。この最後の点が重要で、親は*結論*だけを保持するので、サブエージェントが 40 個のファイルを読んでそこに至ったとしても、メイン会話はきれいなままです。

サブエージェントが手間に見合う理由は 2 つの性質にあります。

- **隔離(Isolation)** — サブエージェントの探索が、メインスレッドのコンテキストを汚しません。
- **並列(Parallelism)** — 独立したサブエージェントが同時に動くので、実時間が短くなります。

## いつ起動すべきか(そしてすべきでないか)

サブエージェントに手を伸ばすのは:

- タスクが**読み込み中心**のとき——「古い認証 API を呼んでいる箇所をすべて見つけて」——で、検索の経路ではなく答えだけが欲しいとき。
- 並行して走らせられる**独立した仕事**があるとき——たとえばモジュール A のテストを書く*かたわら*、モジュール B をリファクタする。
- 独自の指示を持つ**専門的な役割**——レビュアー、エクスプローラー、アーキテクト——が欲しいとき。

タスクが 2 回のツール呼び出しで済む既知の単一編集なら、わざわざ使うことはありません。些細な作業のためにエージェントを立ち上げるオーバーヘッドは割に合いません。

## タスクを並列で回す

カギとなる動き:独立したサブエージェントを**1 つのメッセージで**起動すること。1 つずつ送るとキューに並びます。まとめて送れば、同時に動きます。

私がメインエージェントに与える典型的なプロンプト:

> 「サブエージェントを 3 つ並列で起動して:(1) 決済フローの仕組みを調べて呼び出し連鎖を報告、(2) リポジトリ内のハードコードされたタイムアウトをすべて見つける、(3) `utils/date.ts` のユニットテストを下書き。3 つ全部終わったら報告して。」

それぞれが簡潔なサマリーを返してきます。私はレビューし、次のステップを決めます。重要なのは、3 つの検索がスクロールしていくのを見守る必要が一度もなかったこと——3 つの結論を受け取ったのです。

## サブエージェントに正しい役割を与える

汎用のサブエージェントでも機能しますが、専門化したもののほうが良く機能します。実際には、私はいくつかのパターンに頼っています。

| 役割 | 得意なこと | 返すもの |
|---|---|---|
| エクスプローラー | 不慣れなコードの把握、「X はどこ」 | 特定された答え + 主要なファイルパス |
| アーキテクト | 実装前の機能設計 | 触るべきファイルを含むビルド計画 |
| レビュアー | 差分のバグチェック | 確信度の高い問題のみ |
| 実装者 | 範囲が明確に定義された変更 | テスト済みの変更 |

カスタムエージェント定義をサポートする環境なら、役割ごとに短い指示ファイルを書いておくと、毎回仕事を説明し直さずに済みます。常に「確信度の高い問題だけを報告する」レビュアーエージェントがあれば、細かい指摘をかき分けずに済みます。

## 実際の end-to-end の例

機能フラグの仕組みを追加するとしましょう。私が回す流れはこうです。

1. **アーキテクトのサブエージェント** — 「既存の設定パターンに合う、最小限の機能フラグ層を設計して。作成/変更するファイルとデータフローを返して。」 私が設計図をレビュー。
2. **2 つの実装者サブエージェントを並列で** — 一方はフラグストアを構築、もう一方は React フックを配線。互いの*ファイル*には依存せず、アーキテクトが定義した共有インターフェースにだけ依存するので、同時に走れます。
3. **レビュアーのサブエージェント** — 両方が着地したら、「差分をバグと規約違反についてレビューして。」
4. 最終の結合テストは自分でやる。

かつて長く一本道だった午後が、いくつかの並列バーストになり、私が編集長になります。

## つまずきやすいポイント

- **共有ファイル = 直列化する。** 2 つのサブエージェントが同じファイルを編集するなら、順番に走らせないと衝突します。並列が効くのは、仕事が本当に独立しているときだけです。
- **引き継ぎは具体的に。** サブエージェントはメインスレッドのように途中で確認の質問ができません。あいまいなプロンプトはあいまいなサマリーを生みます。成果物を明記しましょう。
- **委譲した検索を自分で再実行しない。** 検索をサブエージェントに渡したら、待ちましょう。自分でも走らせると token の支出が倍になるだけです。
- **token 使用量に注意。** 大きなリポジトリで並列エージェントが多いと token を速く燃やします。従量課金の統一ゲートウェイ経由なら見張りやすいですが、1 行の修正に 5 つもエージェントを展開しないこと。

## まとめ

サブエージェントは、タスクの*考え方*を変えます。1 本の長い会話の代わりに、仕事を隔離された並列化可能な単位に分解し、あなたはレビュアーになります。小さく始めましょう——読み込み中心の検索を 1 つ委譲し、メインスレッドがどれだけきれいに保たれるかを感じてみてください。次に、独立したファイルで 2 つの実装者を並列で試す。一度しっくりくれば、もう元には戻れません。
