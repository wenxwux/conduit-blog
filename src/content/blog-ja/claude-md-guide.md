Claude Code を使い始めたころ、私はいつも「察してくれない」と感じていました。毎回どのパッケージマネージャを使うか、テストはどう走らせるか、コードスタイルは何かを伝え直す必要があったのです。あとで気づいたのは、こうしたことは `CLAUDE.md` に一度書いてしまえばよいということ——自動で読み取られ、以降は暗黙の了解になります。本記事ではこのファイルの書き方を整理します。

## 一、CLAUDE.md とは何か

`CLAUDE.md` はプロジェクトルートに置く、Claude Code が起動時に自動で読み込む「プロジェクト説明書」です。ここに書いた内容は長期的な背景として、その後のすべての回答に影響します。

平たく言えば、あなたと AI のあいだの「取り決め」です。取り決めが明確なほど、初歩的なミスをせず、何度も訂正させずに済みます。

## 二、何を書くべきか(価値の高い内容)

経験則は、**「知っておかないと間違えること」だけを書く**ことです。プロジェクトのドキュメントを全部持ち込むのではありません。私は次のようなものを書きます。

### 1. 技術スタックとツールチェーン

```
- 言語:TypeScript(strict モード)
- パッケージ管理:pnpm(npm / yarn は使わない)
- フレームワーク:Next.js 14 App Router
- テスト:vitest、実行は `pnpm test`
```

### 2. プロジェクト構成の約束

```
- ビジネスコンポーネントは src/components/ に置く
- ユーティリティ関数は src/lib/ に置く
- components 内で直接データ取得を書かず、src/api/ に統一する
```

### 3. コードスタイルと禁止事項

```
- 関数コンポーネント + hooks を使い、class コンポーネントは書かない
- 新しいサードパーティ依存を勝手に入れず、先に確認を取る
- コミット前に必ず lint を走らせる:`pnpm lint`
```

### 4. よく使うコマンドの早見

```
- 開発起動:pnpm dev
- ビルド:pnpm build
- DB マイグレーション:pnpm db:migrate
```

## 三、何を書くべきでないか(よくある誤解)

- ❌ **長文のビジネスドキュメント**:そういうものは README や wiki に。詰め込んでもコンテキストを浪費するだけです。
- ❌ **頻繁に変わる細かい内容**:書けばメンテが必要になり、古びた説明は無いより悪いです。
- ❌ **シークレットやトークン**:バージョン管理に入るファイルには絶対に書かないこと。
- ❌ **コードと無関係な余計な話**:1 行ごとに token を消費します。簡潔さは美徳です。

## 四、そのままコピーできるテンプレート

```markdown
# プロジェクト説明 (CLAUDE.md)

## 技術スタック
- 言語:
- パッケージ管理:
- フレームワーク:
- テスト:

## プロジェクト構成
-

## コードスタイル
-

## 禁止事項(必ず守る)
- 新しい依存を勝手に入れない
- 変更前に計画を説明する
- コミット前に走らせる:

## よく使うコマンド
- 開発:
- ビルド:
- テスト:
```

空欄を埋めてプロジェクトルートに置けば、それでだいたい使えます。

## 五、メンテして腐らせない

CLAUDE.md は一度書いて終わりではありません。技術スタックが変わったり、コマンドが変わったりしたら、忘れずに同期して更新しましょう。私の習慣は、AI に同じ問題を何度も訂正している自分に気づいたら、そのルールを CLAUDE.md に追記すること——こうすれば訂正は一度だけで、以降は自動で効きます。

## まとめ

よい CLAUDE.md は長くはありませんが、繰り返しの説明を大きく省け、AI の出力をあなたのプロジェクト規約により合わせられます。原則は 3 つだけ。**重要な約束だけを書く、簡潔に保つ、継続的にメンテする**。コンテキストの絞り込みとあわせて使うと、さらに効果的です。
