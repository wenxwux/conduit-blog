## なぜ Claude 全シリーズを知る必要があるのか？

Anthropic の Claude シリーズは今や完全な製品マトリクスを展開しています——軽量高速の Haiku からフラッグシップの Opus まで、その間にメインの Sonnet もあります。多くの開発者の最初の反応は「最強のを使えばいい」ですが、月末の請求書を見ると、80% の token を小さなモデルで処理できるシンプルなタスクに費やしていたことに気づきます。

モデル選定の核心は一言で言えば：**高価な計算力を本当に必要な場所に使う**ことです。この記事では1つの表と複数のシナリオで Claude 全シリーズを整理します。

## 各モデルの適用シナリオ詳解

### Haiku：大量処理の王者

Haiku はシリーズ全体で最も安く、最も速い選択肢です。そのポジショニングは**高スループット・低レイテンシーのバッチタスク**：

- **テキスト分類 / 感情分析**：数十カテゴリの意図認識で、Haiku の精度で十分
- **コンテンツモデレーション**：違反コンテンツのフィルタリング、毎秒大量リクエスト処理
- **シンプルな抽出**：固定フォーマットの文書から構造化フィールドを抽出（氏名、日付、金額）
- **会話サマリー**：カスタマーサポートのチャット履歴を要点に圧縮

```python
# 典型的な使い方：バッチ分類
import anthropic

client = anthropic.Anthropic()

response = client.messages.create(
    model="claude-3-5-haiku-20241022",
    max_tokens=50,
    messages=[{
        "role": "user",
        "content": "以下のチケットを [返金/配送/技術問題/その他] に分類してください：\n'注文して3日経つのにまだ発送されていません'"
    }]
)
```

一言まとめ：**深い思考が不要なタスクは、すべて Haiku に任せる**。

### Sonnet：万能の主力

Claude 4 Sonnet は大半の本番シナリオで最良の選択——品質は Opus に近く、価格はその 1/5：

- **コード生成とレビュー**：関数の作成、デバッグ、テストケース生成
- **長文ドキュメント理解**：契約レビュー、論文サマリー、技術文書 Q&A
- **マルチターン対話**：チャットボット、AI アシスタントのメインエンジン
- **構造化出力**：JSON、表、レポートの生成

### Opus：深度推論の王者

Opus はフラッグシップの中のフラッグシップで、**本当に深い思考が必要な**シナリオに適します：

- **高度な数学 / 論理推論**：コンテストレベルの数学問題、形式的証明
- **多段階システム設計**：アーキテクチャ方案の比較、技術選定分析
- **高難度コード**：アルゴリズム最適化、大規模リファクタリング、クロスモジュールデバッグ
- **高リスク意思決定補助**：法的条項の分析、医療文献レビュー

## 実践モデル選定フローチャート

新しいタスクに直面した際、以下のフローで素早く決定できます：

1. **タスクに深度推論が必要か？**（多段階ロジック、複雑な分析）→ はい → **Opus**
2. **タスクに高品質な理解と生成が必要か？**（コード、長文、創作）→ はい → **Sonnet**
3. **タスクはシンプルで大量処理が必要か？**（分類、抽出、フィルタリング）→ はい → **Haiku**

実際の本番環境では、多くのチームが**モデルルーティング**を行っています——まず軽量な判定器でタスクの難易度を決め、対応するモデルに振り分けます：

```python
def route_to_model(task_complexity: str) -> str:
    """タスクの複雑度に基づいて適切なモデルにルーティング"""
    routing = {
        "simple": "claude-3-5-haiku-20241022",   # 分類、抽出
        "medium": "claude-sonnet-4-20250514",     # コード、対話
        "complex": "claude-opus-4-20250918",      # 深度推論
    }
    return routing.get(task_complexity, "claude-sonnet-4-20250514")
```

## コスト最適化の実践テクニック

モデル選定以外にも、コスト削減テクニックがあります：

- **Prompt Caching**：system prompt が長い場合（大量の背景資料を含むなど）、キャッシュを有効にすれば繰り返し入力の費用を節約可能
- **Batch API**：リアルタイム不要のタスクは Batch API で一括送信、通常 50% の割引
- **max_tokens の制御**：分類タスクは数文字の出力で十分、max_tokens を小さく設定して余分な生成を防止
- **小→大の順で**：開発段階は Haiku で素早くプロンプトを反復、安定したら Sonnet に切り替えて品質検証

| 最適化戦略 | 推定削減 | 適用シナリオ |
|---------|---------|---------|
| モデルダウングレード（Sonnet→Haiku） | 60-75% | シンプルなタスク |
| Batch API | ~50% | 非リアルタイムバッチ処理 |
| Prompt Caching | 最大 90%（キャッシュヒット部分） | 長い system prompt |
| max_tokens 制御 | 10-30% | 短い出力のタスク |

## おわりに

モデル選定に「万能解」はありません。最も賢いやり方は、各モデルのスイートスポットを理解し、システム内にルーティングとフォールバック戦略を組み込むことです。
