手動のコードレビューは価値がありますが遅く、レビュアーは疲れています。解決策は人間を置き換えることではなく——その手前に AI のパスを挟み、明らかなもの(サイレントな失敗、見落としたエッジケース、規約のズレ)を人が見る前に捕まえることです。私はしばらく Codex をその一次レビュアーとして走らせています。実際に持ちこたえているセットアップを紹介します。

## ゴール:2 段階のレビュー

私が使っているメンタルモデル:

1. **ステージ 1 — Codex** が、変更ごとに機械的・パターンレベルの問題を自動で捕まえる。
2. **ステージ 2 — 人間** が、設計・意図・AI が苦手な判断に集中する。

ステージ 1 の狙いは、ステージ 2 を*安くする*ことであり、取り除くことではありません。Codex が 10 件指摘して 8 件が本物なら、人間のレビュアーはずっと良い地点からスタートできます。

## ステップ 1:ローカルのプリコミットパス

問題を捕まえられる最速のタイミングは、コミットされる前です。私は Codex をプリコミットのステップに組み込み、ステージされた差分をレビューさせます。

```bash
# .git/hooks/pre-commit (or via a hook manager)
#!/usr/bin/env bash
DIFF=$(git diff --cached)
if [ -z "$DIFF" ]; then exit 0; fi

echo "$DIFF" | codex review --stdin \
  --focus "bugs,silent-failures,edge-cases" \
  --format markdown > /tmp/codex-review.md

echo "--- Codex review ---"
cat /tmp/codex-review.md
```

これはデフォルトで**非ブロッキング**に保ちます——指摘を表示するだけで、コミットは拒否しません。偽陽性で発火するブロッキングゲートは、人々にそれを回避するよう仕向け、本末転倒になります。

## ステップ 2:焦点を絞ったレビュープロンプト

レビューの品質は、ほぼプロンプトの品質です。あいまいな「このコードをレビューして」からはあいまいな出力しか出ません。私は Codex に厳格なルーブリックを与えます。

> この差分をレビューして。確信のある問題だけを報告して。各項目について、ファイルと行、深刻度(高/中/低)、問題を 1 文で、具体的な修正を示して。優先順位:サイレントな失敗と握りつぶされたエラー、欠けたエラー処理、エッジケース(null/空/境界)、プロジェクト既存の規約への違反。フォーマッタがすでに扱うスタイルにはコメントしない**こと**。差分がきれいならそう言って。

最後の指示が重要です——言うことがないときは黙るよう指示すること、それがノイズを抑える秘訣です。

## ステップ 3:CI に組み込む

ローカルフックも役立ちますが、CI こそチームの習慣になる場所です。私はプルリクエストごとにレビュージョブを走らせ、結果を PR コメントとして投稿します。

```yaml
# .github/workflows/codex-review.yml
name: Codex Review
on: [pull_request]
jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with: { fetch-depth: 0 }
      - name: Run Codex review
        env:
          OPENAI_BASE_URL: ${{ secrets.LLM_BASE_URL }}
          OPENAI_API_KEY: ${{ secrets.LLM_API_KEY }}
        run: |
          git diff origin/${{ github.base_ref }}...HEAD > pr.diff
          codex review --stdin < pr.diff --format markdown > review.md
      - name: Post comment
        run: gh pr comment ${{ github.event.number }} --body-file review.md
        env:
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

実運用で得たいくつかのメモ:

- `OPENAI_BASE_URL` を 1 つのゲートウェイのエンドポイントに向けておくと、同じシークレットがローカルフックと CI の両方で使え、ワークフローに触れずに裏のモデルを差し替えられます。
- `fetch-depth: 0` が重要です——ベースブランチと差分を取るには全履歴が必要です。
- レビューはリポジトリ全体ではなく**差分**に限定しましょう。変更されていないコードのレビューは token を浪費し、本当の指摘を埋もれさせます。

## ステップ 4:シグナルに合わせて調整する

最初の 1 週間は、価値の低いコメントが多すぎるでしょう。締めていきます。

| 症状 | 対処法 |
|---|---|
| 細かい指摘が多すぎる | 「リンタ/フォーマッタがカバーするものは無視」を追加 |
| 意図的なものを指摘する | 関連する規約/CLAUDE.md のコンテキストを与える |
| 本物のバグを見逃す | 明示的なカテゴリ(並行性、off-by-one、リソースリーク)を追加 |
| コメントが長すぎて読めない | 出力を制限:「最大 8 件、深刻度の高い順」 |

回しているダイヤルは**確信度のしきい値**です。確信度の高い問題だけを報告するよう指示すれば、いくつかの軽微な見逃しと引き換えに、チームが実際に読むコメントが得られます。

## 人間に残すべきもの

私は AI のパスにマージの判断をさせることは決してありません。Codex が得意なのは:握りつぶされた例外、未処理の null、コピペバグ、一貫性のないエラー処理、欠けたテストケース。苦手なのは:その機能が*正しい*機能かどうか、その抽象化が将来うまく老いるか、その変更が言葉にされないチームの文脈に合うか。それはレビュアーの仕事で、機械的なパスがすでに走った今、彼らにはそのためのエネルギーがあります。

## まとめ

ここでの勝利は「AI が私のコードをレビューする」ことではなく——層になったパイプラインです:プリコミットが明らかなものを捕まえ、CI がそれをチームの規範にし、人間は本当に重要なところに注意を注ぐ。非ブロッキングのローカルフックから始め、プロンプトを調整し、シグナルが良くなったら CI に昇格させましょう。数週間のうちに AI のパスは目に見えないインフラになり——誰かが開く前に、あなたの PR はきれいになります。
