LLM の機能がプロトタイプから本番に移ると、料金は丸め誤差では済まなくなります。朗報は、支出のほとんどは品質に触れずにエンジニアリングで消せる無駄だということ。私が手を伸ばす 4 つのレバーを、手を伸ばす順に紹介します——もっとも少ない労力でもっとも大きな節約が得られる順だからです。

## 1. プロンプトキャッシュ:同じ token に二度払わない

ほとんどの本番プロンプトには、大きくて安定した接頭辞があります——システムプロンプト、ツール定義、few-shot の例、取得したコンテキストのかたまり——その後に小さな可変部分(ユーザーの実際の質問)が続きます。キャッシュがなければ、その接頭辞全体に毎回フルの入力価格を払うことになります。

プロンプトキャッシュを使えば、プロバイダが接頭辞を保存し、キャッシュヒットには一部の料金だけを課金します。パターンは:

- **安定したものをすべて前に**置く:システム指示、スキーマ、例。
- **可変のユーザー入力を末尾に**置く。
- 接頭辞がキャッシュ可能になるよう境界をマークする。

```python
messages = [
    # --- cacheable prefix: identical across requests ---
    {"role": "system", "content": LONG_SYSTEM_PROMPT},   # 2k tokens
    {"role": "system", "content": TOOL_DEFINITIONS},      # 1k tokens
    {"role": "system", "content": FEW_SHOT_EXAMPLES},     # 3k tokens
    # --- variable suffix: changes every call ---
    {"role": "user", "content": user_question},           # 100 tokens
]
```

同じエージェントを多くのユーザーに提供しているなら、その 6k token の接頭辞が入力コストの大半を占めます——そしてそれをキャッシュすることが、できる中で単一最大の効果を持つ変更です。注意点:キャッシュは失効するので、呼び出しが近接しているときだけキャッシュが報われます。バースト的で大量のトラフィックがもっとも恩恵を受け、1 時間に 1 回の呼び出しはあまり恩恵を受けません。

## 2. バッチ:レイテンシをスループットと交換する

仕事がレイテンシに敏感でないなら——夜間の要約、大量分類、バックログの埋め込み——リアルタイム価格で 1 件ずつリクエストを撃たないこと。バッチ API は多数のリクエストを非同期で、大幅な割引で処理します。

何をバッチ化するかの目安:

- **バッチ化する:**バックグラウンドジョブ、データパイプライン、評価、コンテンツ生成キュー、「数時間以内に終われば OK」なもの全般。
- **バッチ化しない:**チャット、ユーザーが待っているもの全般。

専用のバッチ API がなくても、仕事をまとめられます。1 項目につき 1 回の API 呼び出しではなく、1 回の呼び出しで複数項目を処理させるのです。

```python
# Instead of 50 calls...
for item in items:
    classify(item)

# ...one call that classifies 50 items
classify_batch(items)   # returns a JSON array of 50 labels
```

これは呼び出しごとのオーバーヘッドを削り、キャッシュ可能な接頭辞をより多くの有用な出力で按分できるようにします。出力 token の上限に気をつけ、1 つの不正な応答でバッチ全体を失わないよう、バッチサイズは正気な範囲に保ちましょう。

## 3. モデルルーティング:すべてを大きいモデルに送らない

私がよく見かける最大の過剰支出:*すべての*リクエストをもっとも高性能(でもっとも高価)なモデルにルーティングすること。ほとんどのリクエストにはそれは要りません。

トラフィックを階層化しましょう。

| タスク | モデル層 | 理由 |
|---|---|---|
| 分類、抽出、ルーティング | 小型・安価 | 決定的で簡単、大量 |
| 下書き、要約、簡単な Q&A | 中堅 | 十分良く、ずっと安い |
| 難しい推論、コードアーキテクチャ、あいまいなタスク | 大型 | ここでは価格に値する |

シンプルなルーターが大いに役立ちます。

```python
def pick_model(task):
    if task.type in ("classify", "extract", "route"):
        return "small-model"
    if task.needs_deep_reasoning:
        return "large-model"
    return "mid-model"
```

**カスケード**もできます:まず安いモデルを試し、確信度チェックや検証が失敗したときだけ高価なモデルに格上げする。リクエストの 80% が簡単なワークロードなら、これだけで料金を劇的に削れます。1 つのゲートウェイ経由でルーティングすれば、これは簡単です——SDK やエンドポイントではなく、モデル名の文字列を差し替えるだけです。

## 4. プロンプトそのものを軽くする

入る token も、出る token も、すべてお金です。手早い監査の対象:

- **肥大したシステムプロンプト** — 言い直された指示や死んだ例を削る。短いほうがしばしば*良く*動き、悪くなりません。
- **詰め込みすぎの RAG コンテキスト** — 10 個の文書を投げ込む代わりに、少数のより関連するチャンクを取得する。再ランクして削る。
- **上限のない出力** — `max_tokens` を実際に必要な分だけに設定する。要約器に 4k token の余裕は要りません。
- **冗長なフォーマット** — 捨てることになる散文に包まれた説明ではなく、コンパクトな JSON を求める。

## 組み合わせる

レバーを重ねましょう——複利で効きます。

1. 安定した接頭辞を**キャッシュ**する(繰り返しプロンプトで単一最大の勝利)。
2. ユーザー向けでないものをすべて**バッチ化**する。
3. 簡単なタスクを安いモデルに**ルーティング**し、必要なときだけ格上げする。
4. プロンプトを**削り**、出力に上限を設けて、無駄に払わない。

実際には、私はキャッシュとルーティングから始めます。もっとも少ないリスクでもっとも効果が高いからです。それからバックグラウンドジョブにバッチを重ねます。どれも品質を落とさず——必要のない仕事にフルのリアルタイム価格を払うのをやめるだけです。もっとも安い token は、決して送らない token です。
