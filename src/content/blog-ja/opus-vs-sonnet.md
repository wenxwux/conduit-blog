Claude シリーズの中で、Opus と Sonnet はもっともよく比較される 2 つです。簡単に言えば、Opus 4.8 は「最強の頭脳」、Sonnet 4.6 は「速くて安定した主力」。では日々のタスクに落とし込むと、どちらを使うべきか?本記事ではそのまま使える判断表をお渡しします。

## 一、両者の位置づけの違い

- **Opus 4.8**:推論が最強で、複雑で深い思考を要するタスクに最適。代償は遅さと単価の高さです。
- **Sonnet 4.6**:速度が速く、コストパフォーマンスが高く、総合力のバランスが良い。たいていの日常タスクをきれいにこなします。

核心的なトレードオフは一言です。**Opus は品質をコストで買い、Sonnet はコストを効率で買う**。ほとんどの場面で必要なのは Sonnet で、Opus を出すに値するのは少数の難物だけです。

## 二、場面別の判断表

| タスクの種類 | 推奨モデル | 理由 |
|---|---|---|
| 日常の小機能づくり / バグ修正 | Sonnet 4.6 | 速くて正確、コスパが最高 |
| コメント追加 / テスト作成 / 整形 | Sonnet 4.6 | 簡単な作業に Opus は不要 |
| 大規模なアーキテクチャ設計 / リファクタ方針 | Opus 4.8 | 全体的な推論と判断が必要 |
| 複雑なアルゴリズム / 難しいバグ | Opus 4.8 | 深い推論の強みが明確 |
| ドキュメント作成 / 要件整理 | Sonnet 4.6 | 表現が流暢で十分 |
| 複数ファイルにまたがる大改修 | Opus 4.8 | コンテキスト調整力が高い |
| 簡単な Q&A / API の使い方確認 | Sonnet 4.6 | 応答が速く節約になる |
| 重要な判断 / 失敗できない成果物 | Opus 4.8 | 少しの追加コストで確実さを買う |

## 三、シンプルな判断ルール

私自身の判断ロジックは 2 ステップだけです。

1. **まずデフォルトで Sonnet を使う。** ほとんどのタスクはこれで片付きます。
2. **次のいずれかに当たったときだけ、Opus に格上げする:**
   - Sonnet で 2 回試してもうまくできなかった
   - 複数ファイルの統括や複雑なロジック推論が絡む
   - 今回の成果物が特に重要で、ミスが許されない

言い換えれば、**最初から一番高いものを使わない**こと。安いもので試し、ダメなら格上げする。これで長期的なコストを大きく節約できます。

## 四、切り替え方

Claude Code 内で直接:

```bash
/model
```

そして使いたいモデルを選びます。途中で切り替えることもできます——たとえば、まず Sonnet に方針を出させ、方向が正しいと確認したら Opus に切り替えて、もっとも重要な部分を実行させる、といった具合です。

## 五、混在こそ上級者の使い方

もっとも節約になる使い方は「1 つだけ使う」ことではなく、**組み合わせて使う**ことです。

- 探索・試行錯誤・計画立案の段階 → Sonnet(安いので何度でも試せる)
- 本当に複雑な実装の落とし込み → Opus(一度で仕上げる)
- 仕上げ・テスト追加・ドキュメント作成 → Sonnet に戻す

こうすれば難しいタスクの品質を確保しつつ、簡単な工程でお金を無駄にせずに済みます。

## まとめ

Opus 4.8 と Sonnet 4.6 は「どちらが優れているか」ではなく、「いつどちらを使うか」です。デフォルトは Sonnet、難物だけ Opus、という原則を覚え、上の判断表を組み合わせれば、選択を間違えることはまずありません。モデルを正しく選ぶこと自体が、もっとも大きな節約になります。
