「結局どのモデルを使えばいいの?」これは私がもっともよく聞かれる質問です。答えはいつも「タスク次第」。同じモデルでも、コード作成では神でも、ある場面ではそれほど割に合わないこともあります。本記事では 3 大日常場面に分けて解説し、自分の状況に当てはめられるようにします。

## 一、コードを書く:まず Claude シリーズ

コード作成、バグ修正、リファクタリングにおいて、Claude のコード能力は一貫して頼れます。とくに Claude Code のようなコマンドラインツールと組み合わせれば、プロジェクトを読み、ファイルを直し、テストを走らせる、を一気通貫でこなせます。

具体的な使い分けは:

- **日常の機能、小さな修正、テスト追加** → Sonnet。速くて安く、品質も十分。
- **複雑なアーキテクチャ、難しいバグ、大規模リファクタリング** → Opus。深い推論が強く、追加コストを払う価値があります。

この 2 つの詳細な比較は、[Opus と Sonnet の選び方](/conduit-blog/ja/blog/opus-vs-sonnet/)という記事に書いたので、細かく知りたい方はそちらを。

**一言で言えば:コードを書くならデフォルト Sonnet、難物だけ Opus。**

## 二、文章を書く:十分なら良し、むやみに最強を使わない

マーケティングコピー、推敲、タイトル作成、メール作成——この種のタスクは「推論の深さ」よりも「語感と速度」が求められます。

私の提案は:

- **日常のコピー、書き換え、加筆** → Sonnet のような中堅モデルで十分。応答が速く、コストが低い。
- **長文で、構成が複雑で、トーンの要求が高い内容**(完全なブランドストーリーや長文創作など)→ Opus を使うとよいでしょう。長文の制御と繊細さに優れます。

文章の場面でもっとも犯しやすい間違いは、「タスクが簡単なのに一番高いモデルを使う」こと。製品紹介を書くだけなら、最上位を持ち出す必要はありません。

## 三、分析する:「計算」か「思考」かを見る

データ分析は 2 つのケースに分けます。

### 1. 厳密な推論が必要な分析

たとえば複雑なレポートの読み解き、多次元の要因分析、分析レポートの作成——こうした「深く考える」必要があるものは、**Opus を推奨**します。長い連鎖の推論で安定しています。

### 2. 単純な抽出と要約

たとえばある文章からキー情報を抜き出す、簡単な分類をする、要約を生成する——こういうものは **Sonnet で十分**。最上位を使う必要はなく、速くて節約になります。

> 一点注意:具体的な数値計算が絡むときは、どのモデルでも間違える可能性があります。重要な数字は必ず自分で再確認するか、口頭計算ではなくコードを書いて計算させましょう。

## 四、総合表

| 場面 | サブタイプ | 推奨 |
|---|---|---|
| コードを書く | 日常 / 小修正 | Sonnet |
| コードを書く | 複雑 / リファクタ | Opus |
| 文章を書く | 日常の書き換え | Sonnet |
| 文章を書く | 長文 / 高要求 | Opus |
| 分析する | 深い推論 | Opus |
| 分析する | 抽出・要約 | Sonnet |

## 五、共通の原則

どんな場面でも、私の選択ロジックは同じです。

1. **まず安いもので試す**(Sonnet など)。たいていそれで成功します。
2. **満足できなければ格上げ**して、より強いもの(Opus)へ。
3. **重要な成果物は最初から強いものを**。ここでケチってはいけません。

「デフォルトは安く、必要に応じて格上げ」を習慣にすれば、長期的に節約できる額は少なくなく、それでいて効果はほとんど落ちません。

## まとめ

「最良のモデル」は存在せず、あるのは「もっとも適したモデル」だけです。コードは複雑さで見る、文章は十分なら良し、分析は計算か思考かで見る。タスクとモデルを正しくマッチさせてこそ、良い結果を得ながら、無駄なお金を使わずに済みます。
