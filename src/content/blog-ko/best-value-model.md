"어느 모델이 가장 좋은가?"는 잘못된 질문입니다. 올바른 질문은 "*이 작업에* 가장 가성비가 좋은가"입니다. 가장 강력한 모델은 우리가 실제로 시키는 작업의 절반에는 과잉이고, 가장 저렴한 모델은 나머지 절반에서 조용히 실패하기 때문입니다. 그래서 리더보드를 믿는 대신, 저는 같은 작업 세트를 다섯 모델에 돌려 품질·속도·비용을 함께 살펴봤습니다. 그 프레임워크와 발견한 점을 공유합니다.

## 실험 설정

벤치마크 잡학이 아니라 실제 업무를 반영하는 고정 작업 세트를 사용했습니다.

- **코드:** 명세에서 작은 기능 구현, 그리고 스택 트레이스가 주어진 버그 수정.
- **리팩터링:** 지저분한 200줄 파일 정리.
- **추론:** 검증 가능한 답이 있는 다단계 논리 문제.
- **작문:** 빽빽한 문단을 명료하게 다시 쓰기.
- **추출:** 비정형 텍스트에서 구조화된 JSON 뽑아내기.

각 작업을 플래그십 등급, 중급, 소형/저가 등급을 아우르는 다섯 모델에 돌렸습니다. 출력 품질은 직접 1–5점으로 채점하고, 응답 시간을 재고, token 비용을 기록했습니다. 모델별 정확한 수치는 일부러 공개하지 않습니다. 끊임없이 바뀌고 여러분의 프롬프트에 따라 달라지니까요. 하지만 *패턴*은 안정적이며, 유용한 건 바로 그 패턴입니다.

## "가치"가 실제로 뜻하는 것

지표는 원시 품질이 아니라 달러당 가치입니다.

```
value = quality_score / cost_for_the_task
```

플래그십 모델이 5/5점을 받지만 4.5/5점을 받는 중급 모델의 8배 비용이라면, 대부분의 작업에서 그건 *가성비가 더 나쁩니다*. 마지막 0.5점을 위해 큰 프리미엄을 내는 셈이죠. 그 0.5점이 출시하느냐 못 하느냐의 차이일 때만 의미가 있습니다.

## 일관되게 나타난 패턴

**1. 쉽고 구조화된 작업에서는 등급이 수렴한다.**
추출과 분류는 다섯 모델 모두 거의 동일하게 보였습니다. 가장 저렴한 모델조차 깔끔하고 올바른 JSON을 만들어냈죠. 여기서 플래그십에 돈을 내는 건 순수한 낭비입니다. 소형 모델이 압도적인 가성비 승자입니다.

**2. 어려운 추론에서는 격차가 진짜다.**
다단계 논리 문제와 까다로운 버그 수정은 모델들을 갈라놓았습니다. 플래그십과 강한 중급 모델은 해냈지만, 저가 모델들은 자신만만하게 틀린 답을 내놓았습니다. 바로 여기가 프리미엄이 정당화되는 지점입니다. *그럴듯해 보이는* 틀린 답은 세상에서 가장 비싼 출력이니까요.

**3. 중급이 조용한 가성비 챔피언이다.**
초안 작성, 리팩터링, 표준 코딩 전반에서 좋은 중급 모델은 플래그십 품질의 약 90%를 가격의 일부로 달성했습니다. 일상 업무의 대부분에서 이게 스위트 스폿입니다. 저는 여기를 기본값으로 두고, 정말 어려운 걸 만났을 때만 격상합니다.

**4. 속도가 생각보다 더 중요하다.**
인터랙티브 코딩에서는 품질이 약간 낮아도 눈에 띄게 빠른 모델이 종종 *함께 일하기에 더 좋게* 느껴졌습니다. 피드백 루프가 더 짧고, 분당 반복이 더 많죠. 사람이 루프 안에 있을 때는 지연 시간도 가치의 일부입니다.

## 제가 실제로 쓰는 결정표

| 작업 | 가성비 최적 선택 | 이유 |
|---|---|---|
| 분류 / 추출 / 라우팅 | 소형/저가 | 품질이 수렴; 비용이 지배적 |
| 초안, 요약, 표준 코드 | 중급 | 품질 약 90%, 큰 비용 절감 |
| 알려진 파일 리팩터링 | 중급 | 충분히 기계적; 프리미엄 아낌 |
| 어려운 버그 / 모호한 설계 | 플래그십 | 여기선 0.5점이 값어치 함 |
| 인터랙티브 페어 코딩 | 빠른 중급 | 루프 속도가 한계 품질을 이김 |

## 직접 테스트하는 법 (제 걸 믿지 마세요)

모델은 매달 바뀝니다. 유일하게 의미 있는 비교는 *여러분의* 프롬프트에서의 비교입니다. 이렇게 하세요.

1. 실제 업무를 대표하는 작업 5~10개를 고른다.
2. 프롬프트를 한 번 작성하고, 후보 모델들에 그대로 돌린다.
3. 품질을 블라인드로 채점한다(어느 모델이 무엇을 만들었는지 가린다).
4. 작업별 비용과 지연 시간을 기록한다.
5. `quality / cost`를 계산하되, 전체가 아니라 작업 유형별로 본다.

실무적 걸림돌은 보통 배관 작업입니다. 제공자마다 다른 SDK, 키, 엔드포인트가 A/B 테스트를 고통스럽게 만들죠. 저는 다섯 모델 모두를 하나의 게이트웨이, 단일 엔드포인트로 돌려서, 모델 전환이 요청 안의 문자열 하나를 바꾸는 일이 됩니다. 그러면 "어느 모델이 가장 가성비 좋은가"가 연구 프로젝트에서, 새 모델이 나올 때마다 다시 돌릴 수 있는 5분짜리 실험으로 바뀝니다.

## 핵심 요점

단 하나의 최고 모델은 없습니다. 작업별로 가장 *잘 맞는* 모델이 있을 뿐입니다. 비싼 실수는 잘못된 플래그십을 고르는 게 아니라, 쉽고 대량인 작업을 애초에 플래그십에 보내는 것입니다. 강한 중급 모델을 기본값으로 두고, 구조화된 작업은 저가로 내리고, 프리미엄 값을 하는 어려운 추론에만 플래그십으로 격상하세요. 여러분 자신의 작업에서 달러당 가치를 측정하고, 모델이 바뀌면 다시 측정하세요. 모델은 반드시 바뀌니까요.
