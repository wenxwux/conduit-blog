Claude Code에서 얻은 가장 큰 속도 향상은 더 빠른 모델이 아니라, 모든 걸 한 대화에서 처리하길 그만두는 법을 배운 것이었습니다. 서브에이전트는 자기 완결적인 작업 덩어리를, 각자의 컨텍스트 윈도로 실행되는 별도 에이전트에게 넘긴 뒤 결과를 보고받게 해줍니다. 제대로 하면 "긴 스레드 하나를 돌보는" 것에서 "다섯 작업을 띄워 놓고 결과를 검토하는" 것으로 바뀝니다.

제가 실제로 어떻게 쓰는지 소개합니다.

## 서브에이전트란 정말 무엇인가

서브에이전트는 메인 에이전트가 띄우는 새로운 Claude Code 인스턴스입니다. 자기만의 컨텍스트 윈도를 갖고, 한 가지 일을 하며, 도중에 읽은 원시 파일 덤프가 아니라 최종 요약을 반환합니다. 마지막 부분이 중요합니다. 부모는 *결론*만 보관하므로, 서브에이전트가 40개 파일을 읽었더라도 메인 대화는 깔끔하게 유지됩니다.

서브에이전트가 수고할 가치가 있게 만드는 두 가지 속성:

- **격리** — 서브에이전트의 탐색이 메인 스레드의 컨텍스트를 오염시키지 않습니다.
- **병렬성** — 독립적인 서브에이전트가 동시에 실행되어 실제 소요 시간이 줄어듭니다.

## 언제 띄울까 (그리고 언제 띄우지 말까)

서브에이전트를 쓰세요:

- 작업이 **읽기 중심**일 때 — "옛 인증 API를 호출하는 모든 곳 찾기" — 그리고 검색 과정이 아니라 답만 원할 때.
- 나란히 실행할 수 있는 **독립적인 작업**이 있을 때 — 예: 모듈 B를 리팩터링하는 *동안* 모듈 A의 테스트를 작성.
- 자기만의 지시를 가진 **특화된 역할**을 원할 때 — 리뷰어, 탐색가, 아키텍트.

두 번의 도구 호출로 할 수 있는 단일한 알려진 수정이라면 굳이 쓰지 마세요. 사소한 작업에는 에이전트를 띄우는 오버헤드가 값어치를 못 합니다.

## 작업을 병렬로 실행하기

핵심 동작: 독립적인 서브에이전트를 **한 메시지에** 띄우세요. 하나씩 보내면 큐에 쌓입니다. 함께 보내면 동시에 실행됩니다.

제가 메인 에이전트에게 주는 전형적인 프롬프트:

> "서브에이전트 셋을 병렬로 띄워: (1) 결제 흐름이 어떻게 작동하는지 탐색하고 호출 체인을 보고, (2) 저장소의 모든 하드코딩된 타임아웃을 찾고, (3) `utils/date.ts`의 단위 테스트 초안을 작성. 셋 다 끝나면 보고해."

각자 짧은 요약을 들고 돌아옵니다. 저는 검토한 뒤 다음 단계를 정합니다. 결정적으로 세 검색이 스크롤되며 지나가는 걸 지켜볼 필요가 전혀 없었습니다. 결론 세 개를 받았으니까요.

## 서브에이전트에게 올바른 역할 부여하기

범용 서브에이전트도 작동하지만, 특화된 쪽이 더 잘 작동합니다. 실무에서 저는 몇 가지 패턴에 기댑니다.

| 역할 | 적합한 일 | 반환하는 것 |
|---|---|---|
| 탐색가 | 낯선 코드 매핑, "X가 어디 있나" | 찾아낸 답 + 핵심 파일 경로 |
| 아키텍트 | 코딩 전 기능 설계 | 손댈 파일이 포함된 빌드 계획 |
| 리뷰어 | 변경분의 버그 점검 | 고신뢰 이슈만 |
| 구현자 | 범위가 명확한 변경 | 테스트된 변경분 |

설정이 커스텀 에이전트 정의를 지원한다면, 역할별로 짧은 지시 파일을 작성해 매번 일을 다시 설명하지 않도록 하세요. 늘 "고신뢰 이슈만 보고"하는 리뷰어 에이전트는 사소한 트집을 헤집고 다닐 일을 덜어 줍니다.

## 실제 엔드 투 엔드 예시

피처 플래그 시스템을 추가한다고 합시다. 제가 돌릴 흐름은 이렇습니다.

1. **아키텍트 서브에이전트** — "기존 설정 패턴에 맞는 최소한의 피처 플래그 레이어를 설계. 생성/수정할 파일과 데이터 흐름을 반환." 저는 청사진을 검토합니다.
2. **병렬 구현자 서브에이전트 둘** — 하나는 플래그 스토어를 만들고, 하나는 React 훅을 연결. 서로의 *파일*에 의존하지 않고 아키텍트가 정의한 공유 인터페이스에만 의존하므로 함께 실행됩니다.
3. **리뷰어 서브에이전트** — 둘 다 안착하면 "변경분에서 버그와 규약 위반을 검토."
4. 마지막 통합 테스트는 제가 직접 합니다.

예전엔 길고 단일 스레드였던 오후가, 제가 편집장 역할을 맡는 몇 번의 병렬 폭발로 바뀝니다.

## 사람들이 자주 당하는 함정

- **공유 파일 = 직렬화하라.** 두 서브에이전트가 같은 파일을 수정하면, 순차적으로 실행하지 않으면 충돌이 납니다. 병렬성은 작업이 진정으로 독립적일 때만 도움이 됩니다.
- **인계할 때 구체적으로 하라.** 서브에이전트는 메인 스레드처럼 작업 도중에 명확화 질문을 던질 수 없습니다. 모호한 프롬프트는 모호한 요약을 낳습니다. 산출물을 명확히 적으세요.
- **위임한 검색을 다시 돌리지 마라.** 조회를 서브에이전트에게 넘겼으면 기다리세요. 직접 또 돌리면 token 지출만 두 배가 됩니다.
- **token 사용량을 주시하라.** 큰 저장소에서 많은 병렬 에이전트는 token을 빠르게 태웁니다. 사용량 기반 과금의 통합 게이트웨이를 경유한다면 주시하기 쉽지만, 한 줄짜리 수정에 다섯 에이전트를 펼치지는 마세요.

## 마무리

서브에이전트는 작업을 *생각하는* 방식을 바꿉니다. 긴 대화 하나 대신, 일을 격리되고 병렬화 가능한 단위로 분해하고 여러분은 리뷰어가 됩니다. 작게 시작하세요. 읽기 중심 검색 하나를 위임하고 메인 스레드가 얼마나 깔끔하게 유지되는지 보세요. 그다음 독립적인 파일에 병렬 구현자 둘을 시도해 보세요. 한번 감을 잡으면 되돌아가지 않을 겁니다.
