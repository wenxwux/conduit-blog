처음 Claude Code를 쓸 때 저는 그것이 늘 "눈치가 없다"고 느꼈습니다. 매번 어떤 패키지 매니저를 쓰는지, 테스트는 어떻게 돌리는지, 코드 스타일이 무엇인지 다시 알려줘야 했죠. 나중에야 알았습니다. 이런 것들은 `CLAUDE.md`에 한 번만 작성하면 된다는 걸요. 모델이 자동으로 읽어 들이고, 그때부터는 말하지 않아도 통합니다. 이 글에서는 이 파일을 어떻게 써야 하는지 명확히 설명합니다.

## 1. CLAUDE.md란 무엇인가

`CLAUDE.md`는 프로젝트 루트에 두는, Claude Code가 시작할 때 자동으로 불러오는 "프로젝트 설명서"입니다. 여기에 적은 내용은 장기 배경으로 작용해, 이후의 모든 답변에 영향을 줍니다.

쉽게 말해, 여러분과 AI 사이의 "약속"입니다. 약속을 분명히 쓸수록 AI는 저수준 실수를 덜 하고, 여러분이 거듭 바로잡을 일도 줄어듭니다.

## 2. 무엇을 넣어야 하나 (고가치 내용)

경험적으로는 **"반드시 알아야 실수하지 않는 것"만 넣는 것**이 좋습니다. 모든 프로젝트 문서를 다 옮겨오는 게 아니라요. 저는 이런 부류를 넣습니다.

### 1. 기술 스택과 도구 체인

```
- 언어: TypeScript(엄격 모드)
- 패키지 관리: pnpm(npm / yarn 사용 금지)
- 프레임워크: Next.js 14 App Router
- 테스트: vitest, `pnpm test` 실행
```

### 2. 프로젝트 구조 약속

```
- 비즈니스 컴포넌트는 src/components/에
- 유틸 함수는 src/lib/에
- components 안에서 직접 데이터 요청하지 말고, 일괄적으로 src/api/ 경유
```

### 3. 코드 스타일과 레드라인

```
- 함수형 컴포넌트 + hooks 사용, class 컴포넌트 작성 금지
- 새 서드파티 의존성 도입 금지, 먼저 나에게 물어볼 것
- 커밋 전 반드시 lint 실행: `pnpm lint`
```

### 4. 자주 쓰는 명령 빠른 참조

```
- 개발 시작: pnpm dev
- 빌드: pnpm build
- DB 마이그레이션: pnpm db:migrate
```

## 3. 무엇을 넣지 말아야 하나 (흔한 오해)

- ❌ **장문의 비즈니스 문서**: 그런 건 README나 wiki에 두세요. 여기 욱여넣으면 컨텍스트만 낭비됩니다.
- ❌ **자주 바뀌는 세부 사항**: 적으면 유지보수해야 합니다. 낡은 설명은 없느니만 못합니다.
- ❌ **비밀 키, token**: 버전 관리에 들어가는 어떤 파일에도 절대 쓰지 마세요.
- ❌ **코드와 무관한 군말**: 한 줄 한 줄이 token을 차지합니다. 간결함이 미덕입니다.

## 4. 바로 복사해 쓰는 템플릿

```markdown
# 프로젝트 설명 (CLAUDE.md)

## 기술 스택
- 언어:
- 패키지 관리:
- 프레임워크:
- 테스트:

## 프로젝트 구조
-

## 코드 스타일
-

## 레드라인 (반드시 준수)
- 임의로 새 의존성 도입 금지
- 변경 전 먼저 계획 설명
- 커밋 전 실행:

## 자주 쓰는 명령
- 개발:
- 빌드:
- 테스트:
```

빈칸을 채워 프로젝트 루트에 두면 기본적으로 바로 쓸 수 있습니다.

## 5. 유지보수하기, 썩게 두지 말기

CLAUDE.md는 한 번 쓰고 끝이 아닙니다. 기술 스택이 바뀌거나 명령이 달라지면 동기화해 업데이트하세요. 제 습관은 이렇습니다. AI에게 같은 문제를 또 반복해서 바로잡고 있는 자신을 발견할 때마다, 그 규칙을 CLAUDE.md에 추가합니다. 그러면 그 문제는 한 번만 바로잡으면 되고, 이후로는 자동으로 적용됩니다.

## 마무리

좋은 CLAUDE.md는 길지 않지만, 거듭 설명하는 수고를 크게 덜어 주고, AI 출력이 여러분의 프로젝트 규범에 더 잘 들어맞게 합니다. 원칙은 세 가지뿐입니다. **핵심 약속만 쓰기, 간결하게 유지하기, 꾸준히 유지보수하기.** 컨텍스트를 간결히 하는 기법과 함께 쓰면 효과가 더 좋습니다.
