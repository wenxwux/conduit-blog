LLM 기능이 프로토타입에서 프로덕션으로 넘어가면, 청구서는 더 이상 반올림 오차가 아닙니다. 좋은 소식은 지출의 대부분이 품질을 건드리지 않고 엔지니어링으로 없앨 수 있는 낭비라는 점입니다. 제가 손대는 네 가지 레버를, 가장 적은 노력으로 가장 큰 절약을 주는 순서대로 소개합니다.

## 1. 프롬프트 캐싱: 같은 token에 거듭 돈 내지 않기

대부분의 프로덕션 프롬프트에는 크고 안정적인 접두부가 있습니다 — 시스템 프롬프트, 도구 정의, few-shot 예시, 검색된 컨텍스트 덩어리 — 그 뒤에 작은 가변부(사용자의 실제 질문)가 붙죠. 캐싱이 없으면 그 전체 접두부에 대해 매 호출마다 입력 전액을 냅니다.

프롬프트 캐싱은 제공자가 접두부를 저장하고 캐시 적중 시 일부만 청구하게 해줍니다. 패턴:

- **안정된 모든 것을 앞에** 두기: 시스템 지시, 스키마, 예시.
- **가변적 사용자 입력은 끝에** 두기.
- 접두부가 캐시 가능하도록 경계를 표시하기.

```python
messages = [
    # --- cacheable prefix: identical across requests ---
    {"role": "system", "content": LONG_SYSTEM_PROMPT},   # 2k tokens
    {"role": "system", "content": TOOL_DEFINITIONS},      # 1k tokens
    {"role": "system", "content": FEW_SHOT_EXAMPLES},     # 3k tokens
    # --- variable suffix: changes every call ---
    {"role": "user", "content": user_question},           # 100 tokens
]
```

같은 에이전트를 많은 사용자에게 서빙한다면, 그 6k-token 접두부가 입력 비용의 대부분이며, 그것을 캐싱하는 것이 할 수 있는 단일 최고 레버리지 변경입니다. 함정: 캐시는 만료되므로, 호출이 서로 가까이 들어올 때만 캐싱이 이득입니다. 폭발적이고 대량인 트래픽이 가장 큰 혜택을 보고, 시간당 한 번 호출은 덜합니다.

## 2. 배칭: 지연 시간을 처리량과 맞바꾸기

작업이 지연에 민감하지 않다면 — 야간 요약, 대량 분류, 백로그 임베딩 — 실시간 가격으로 요청을 하나씩 쏘지 마세요. 배치 API는 많은 요청을 비동기로, 큰 할인가에 처리합니다.

무엇을 배칭할지 판단 기준:

- **배칭하라:** 백그라운드 작업, 데이터 파이프라인, 평가, 콘텐츠 생성 큐, "몇 시간 내 완료"면 괜찮은 모든 것.
- **배칭하지 마라:** 채팅, 사용자가 기다리는 모든 것.

전용 배치 API가 없어도 작업을 묶을 수 있습니다. 항목당 한 번씩 API를 호출하는 대신, 모델에게 한 호출에서 여러 항목을 처리하라고 시키세요.

```python
# Instead of 50 calls...
for item in items:
    classify(item)

# ...one call that classifies 50 items
classify_batch(items)   # returns a JSON array of 50 labels
```

이것은 호출당 오버헤드를 줄이고, 캐시 가능한 접두부를 더 많은 유용한 출력에 분산시킵니다. 출력 token 한도를 주시하고, 단일한 잘못된 응답이 배치 전체를 날리지 않도록 배치 크기를 합리적으로 유지하세요.

## 3. 모델 라우팅: 모든 걸 큰 모델에 보내지 않기

제가 가장 흔히 보는 과지출: *모든* 요청을 가장 강력한(그리고 가장 비싼) 모델로 라우팅하는 것입니다. 대부분의 요청은 그것이 필요 없습니다.

트래픽을 등급화하세요:

| 작업 | 모델 등급 | 이유 |
|---|---|---|
| 분류, 추출, 라우팅 | 소형/저가 | 결정적, 쉬움, 대량 |
| 초안, 요약, 간단한 Q&A | 중급 | 충분히 좋고, 훨씬 저렴 |
| 어려운 추론, 코드 아키텍처, 모호한 작업 | 대형 | 여기선 값어치 함 |

간단한 라우터가 큰 역할을 합니다.

```python
def pick_model(task):
    if task.type in ("classify", "extract", "route"):
        return "small-model"
    if task.needs_deep_reasoning:
        return "large-model"
    return "mid-model"
```

**캐스케이드**도 가능합니다: 먼저 저가 모델을 시도하고, 신뢰도 점검이나 검증이 실패할 때만 비싼 모델로 격상하세요. 요청의 80%가 쉬운 워크로드라면, 이것만으로 청구서를 극적으로 줄일 수 있습니다. 단일 게이트웨이를 경유하면 이게 사소해집니다 — SDK나 엔드포인트가 아니라 모델 이름 문자열을 바꾸면 됩니다.

## 4. 프롬프트 자체를 슬림하게

들어가는 모든 token, 나오는 모든 token이 돈입니다. 빠른 점검 대상:

- **비대한 시스템 프롬프트** — 반복된 지시와 죽은 예시를 쳐내세요. 더 짧은 것이 종종 *더* 잘 작동합니다.
- **과적된 RAG 컨텍스트** — 문서 열 개를 쏟아붓는 대신, 더 적고 더 관련 있는 청크를 검색하세요. 재순위화하고 잘라내세요.
- **상한 없는 출력** — `max_tokens`를 실제로 필요한 만큼으로 설정하세요. 요약기는 4k token의 여유가 필요 없습니다.
- **장황한 포맷** — 버릴 산문으로 감싼 설명이 아니라, 간결한 JSON을 요청하세요.

## 종합하기

레버를 쌓으세요 — 복리로 작용합니다.

1. 안정된 접두부를 **캐싱**한다(반복 프롬프트에서 가장 큰 단일 승리).
2. 사용자 대면이 아닌 모든 것을 **배칭**한다.
3. 쉬운 작업을 저가 모델로 **라우팅**하고, 필요할 때만 격상한다.
4. 프롬프트를 **다듬고** 출력을 제한해, 군더더기에 돈 내지 않는다.

실무에서 저는 캐싱과 라우팅으로 시작합니다. 가장 적은 위험으로 가장 큰 레버리지이기 때문이죠. 그다음 백그라운드 작업에 배칭을 더합니다. 이 중 무엇도 품질을 떨어뜨리지 않습니다 — 그저 필요 없는 작업에 실시간 전액을 내는 걸 멈추게 할 뿐입니다. 가장 저렴한 token은 애초에 보내지 않는 token입니다.
