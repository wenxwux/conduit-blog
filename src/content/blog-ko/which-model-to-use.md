"도대체 어떤 모델을 써야 하나요?" 제가 가장 많이 받는 질문입니다. 답은 언제나 "작업에 따라 다르다"입니다. 같은 모델이라도 코드 작성에서는 신처럼 뛰어나지만, 어떤 상황에서는 그만큼 합리적이지 않을 수 있습니다. 이 글에서는 일상의 세 가지 주요 상황으로 나눠, 여러분이 자신의 경우에 맞게 고를 수 있도록 돕습니다.

## 1. 코드 작성: Claude 계열이 1순위

코드 작성, 버그 수정, 리팩터링에서 Claude의 코드 능력은 줄곧 강력합니다. 특히 Claude Code 같은 커맨드라인 도구와 함께 쓰면 프로젝트 읽기, 파일 수정, 테스트 실행까지 일사천리입니다.

구체적으로 나누면:

- **일상 기능, 소소한 수정, 테스트 보강** → Sonnet. 빠르고 저렴하며 품질도 충분합니다.
- **복잡한 아키텍처, 까다로운 버그, 대규모 리팩터링** → Opus. 깊은 추론이 더 강해 비용을 더 쓸 가치가 있습니다.

이 둘의 세밀한 비교는 [Opus vs Sonnet 어떻게 고를까](/conduit-blog/ko/blog/opus-vs-sonnet/)에 따로 썼으니, 디테일을 파고들고 싶다면 그 글을 보세요.

**한마디로: 코드 작성은 기본값 Sonnet, 난제만 Opus.**

## 2. 카피 작성: 충분하면 됐지, 무턱대고 최강을 쓰지 말기

마케팅 카피 작성, 윤문, 제목 뽑기, 이메일 쓰기 — 이런 작업은 "추론 깊이"에 대한 요구는 높지 않고, "어감과 속도"에 대한 요구가 더 높습니다.

제 제안:

- **일상 카피, 리라이트, 확장** → Sonnet 같은 중급 모델로 충분합니다. 응답이 빠르고 비용이 낮습니다.
- **장문, 구조가 복잡하거나 톤 요구가 높은 콘텐츠**(예: 완전한 브랜드 스토리, 장문 창작) → Opus를 쓸 수 있습니다. 장문에 대한 장악력과 섬세함이 더 좋습니다.

카피 상황에서 가장 저지르기 쉬운 실수는 "작업은 단순한데 가장 비싼 모델을 쓰는 것"입니다. 제품 소개 한 편 쓰는 데 최고 사양을 동원할 필요는 없습니다.

## 3. 분석: "계산"인지 "사고"인지를 보기

데이터 분석은 두 경우로 나뉩니다.

### 1. 엄밀한 추론이 필요한 분석

복잡한 리포트 해석, 다차원 귀인, 분석 보고서 작성 같은 것은 모델이 "깊이 생각해야" 합니다. 이런 경우 **Opus를 추천**합니다. 긴 사슬 추론에서 더 안정적입니다.

### 2. 단순한 추출과 요약

텍스트에서 핵심 정보 뽑기, 간단한 분류, 요약 생성 같은 것은 **Sonnet으로 충분**합니다. 최고 사양을 쓸 필요 없이 빠르고 절약됩니다.

> 한 가지 당부: 구체적인 수치 계산이 얽히면 어떤 모델이든 틀릴 수 있습니다. 핵심 숫자는 반드시 직접 재확인하거나, 모델이 코드를 작성해 계산하게 하세요. 암산으로 시키지 마세요.

## 4. 종합표

| 상황 | 하위 유형 | 추천 |
|---|---|---|
| 코드 작성 | 일상 / 소소한 수정 | Sonnet |
| 코드 작성 | 복잡 / 리팩터링 | Opus |
| 카피 작성 | 일상 리라이트 | Sonnet |
| 카피 작성 | 장문 / 고요구 | Opus |
| 분석 | 깊은 추론 | Opus |
| 분석 | 추출·요약 | Sonnet |

## 5. 일반 원칙

어떤 상황이든 제 선택 로직은 같습니다.

1. **먼저 저렴한 걸로 시도**(Sonnet 같은). 대개는 그걸로 됩니다.
2. **만족스럽지 않으면 더 강한 걸로 업그레이드**(Opus).
3. **중요한 산출물은 처음부터 강한 걸로.** 이 돈은 아끼면 안 됩니다.

"기본값은 저렴하게, 필요에 따라 업그레이드"를 습관으로 들이면, 장기적으로 아끼는 게 적지 않으면서 효과는 거의 그대로입니다.

## 마무리

"가장 좋은 모델"은 없고 "가장 적합한 모델"만 있습니다. 코드 작성은 복잡도를 보고, 카피 작성은 충분하면 됐고, 분석은 계산인지 사고인지를 봅니다. 작업과 모델을 올바로 매칭해야 좋은 결과도 얻고 헛돈도 안 씁니다.
