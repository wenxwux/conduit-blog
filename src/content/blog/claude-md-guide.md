刚用 Claude Code 时,我总觉得它"不懂事":每次都要重新告诉它用什么包管理器、测试怎么跑、代码风格是什么。后来才发现,这些完全可以一次性写进 `CLAUDE.md`——它会自动读取,从此心照不宣。这篇讲清楚这个文件该怎么写。

## 一、CLAUDE.md 是什么

`CLAUDE.md` 放在项目根目录,是 Claude Code 启动时会自动加载的"项目说明书"。你写在里面的内容会作为长期背景,影响它后续的每一次回答。

说白了,它就是你和 AI 之间的"约定"。约定写得越清楚,它越不会犯低级错误、越不用你反复纠正。

## 二、该放什么(高价值内容)

经验是:**只放它"必须知道才不会做错"的东西**,而不是把所有项目文档都搬进来。我会放这几类:

### 1. 技术栈与工具链

```
- 语言:TypeScript(严格模式)
- 包管理:pnpm(不要用 npm / yarn)
- 框架:Next.js 14 App Router
- 测试:vitest,运行 `pnpm test`
```

### 2. 项目结构约定

```
- 业务组件放 src/components/
- 工具函数放 src/lib/
- 不要在 components 里直接写数据请求,统一走 src/api/
```

### 3. 代码风格与红线

```
- 用函数式组件 + hooks,不写 class 组件
- 不要引入新的第三方依赖,先问我
- 提交前必须跑 lint:`pnpm lint`
```

### 4. 常用命令速查

```
- 启动开发:pnpm dev
- 构建:pnpm build
- 数据库迁移:pnpm db:migrate
```

## 三、不该放什么(常见误区)

- ❌ **大段业务文档**:那些放 README 或 wiki,塞进来只会浪费上下文。
- ❌ **会频繁变动的细节**:写了就要维护,过时的说明比没有更糟。
- ❌ **密钥、token**:绝对不要写进任何会进版本库的文件。
- ❌ **跟代码无关的废话**:每一行都占 token,精简是美德。

```markdown
- 语言:
- 包管理:
- 框架:
- 测试:

## 项目结构
-

## 代码风格
-

## 红线(必须遵守)
- 不要擅自引入新依赖
- 改动前先说明计划
- 提交前跑:

## 常用命令
- 开发:
- 构建:
- 测试:
```

把空白填上,放进项目根目录,基本就能用了。

## 五、维护它,别让它腐烂

CLAUDE.md 不是写一次就完事。技术栈换了、命令变了,记得同步更新。我的习惯是:每次发现自己又在重复纠正 AI 同一个问题,就把那条规则补进 CLAUDE.md——这样问题只纠正一次,以后自动生效。

## 小结

一份好的 CLAUDE.md 不长,但能省下大量反复解释的口舌,也能让 AI 输出更贴合你的项目规范。原则就三条:**只写关键约定、保持精简、持续维护**。配合精简上下文一起用,效果更好。
