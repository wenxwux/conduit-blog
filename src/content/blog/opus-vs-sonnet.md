Claude 系列里,Opus 和 Sonnet 是最常被拿来对比的两个。简单说:Opus 4.8 是"最强大脑",Sonnet 4.6 是"快而稳的主力"。但具体到每天的任务,到底该用哪个?这篇给你一张能直接照着用的决策表。

## 一、两者的定位差异

- **Opus 4.8**:推理最强、最适合复杂、需要深度思考的任务。代价是更慢、单价更高。
- **Sonnet 4.6**:速度快、性价比高、综合能力很均衡。大多数日常任务它都能漂亮地完成。

核心权衡就一句话:**Opus 用质量换成本,Sonnet 用成本换效率**。绝大多数时候,你需要的是 Sonnet;只有少数硬骨头,才值得动用 Opus。

## 二、场景化决策表

| 任务类型 | 推荐模型 | 理由 |
|---|---|---|
| 日常写小功能 / 改 bug | Sonnet 4.6 | 又快又准,性价比最高 |
| 补注释 / 写测试 / 改格式 | Sonnet 4.6 | 简单活,没必要上 Opus |
| 大型架构设计 / 重构方案 | Opus 4.8 | 需要全局推理与权衡 |
| 复杂算法 / 难调的 bug | Opus 4.8 | 深度推理优势明显 |
| 写文档 / 整理需求 | Sonnet 4.6 | 表达流畅,够用 |
| 跨多文件的大改动 | Opus 4.8 | 上下文协调能力更强 |
| 快速问答 / 查 API 用法 | Sonnet 4.6 | 响应快,省钱 |
| 关键决策 / 不能出错的产出 | Opus 4.8 | 多花点钱买稳妥 |

## 三、一个简单的判断法则

我自己的决策逻辑就两步:

1. **先默认用 Sonnet。** 大部分任务它都能搞定。
2. **遇到下面任一情况,才升级到 Opus:**
   - Sonnet 试了两次都没做对
   - 任务需要跨多个文件统筹、或牵涉复杂逻辑推理
   - 这次产出特别重要,不容许出错

换句话说:**别一上来就用最贵的**。先用便宜的试,不行再升级,这样长期成本能省一大截。

## 四、怎么切换

在 Claude Code 里直接:

```bash
/model
```

然后选你要的模型。也可以中途切——比如先用 Sonnet 让它出方案,确认方向对了,再切 Opus 让它执行最关键的部分。

## 五、混合使用才是高手玩法

最省的用法不是"只用一个",而是**组合使用**:

- 探索、试错、列计划阶段 → Sonnet(便宜,可以多试几次)
- 真正落地复杂实现 → Opus(一次到位)
- 收尾、补测试、写文档 → 切回 Sonnet

这样既保证了硬任务的质量,又不会在简单环节浪费钱。

## 小结

Opus 4.8 和 Sonnet 4.6 不是"谁更好",而是"什么时候用谁"。记住默认 Sonnet、硬骨头才上 Opus 这条原则,再配合上面的决策表,基本就不会选错了。模型选对了,本身就是最大的一笔省钱。
