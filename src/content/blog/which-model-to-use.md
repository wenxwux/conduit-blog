"到底该用哪个模型?"这是我被问得最多的问题。答案永远是"看任务"。同一个模型,在写代码上是神,在某些场景可能就没那么划算。这篇按三大日常场景拆开讲,帮你对号入座。

## 一、写代码:首选 Claude 系列

写代码、改 bug、重构,Claude 的代码能力一直很能打,尤其配合 Claude Code 这种命令行工具,读项目、改文件、跑测试一条龙。

具体怎么分:

- **日常功能、小改动、补测试** → Sonnet。又快又省,质量足够。
- **复杂架构、难调的 bug、大规模重构** → Opus。深度推理更强,值得为它多花钱。

这两者的细致对比,我写过一篇 [Opus vs Sonnet 怎么选](/conduit-blog/blog/opus-vs-sonnet/),想抠细节可以看那篇。

**一句话:写代码默认 Sonnet,硬骨头才上 Opus。**

## 二、写文案:够用就好,别盲目上最强

写营销文案、改润色、起标题、写邮件——这类任务对"推理深度"要求不高,对"语感和速度"要求更高。

我的建议:

- **日常文案、改写、扩写** → Sonnet 这类中端模型完全够用,响应快、成本低。
- **需要长篇、结构复杂、调性要求高的内容**(比如完整的品牌故事、长文创作)→ 可以上 Opus,它对长文的把控和细腻度更好。

文案场景最容易犯的错就是"任务很简单却用了最贵的模型"。写个产品介绍而已,没必要动用顶配。

## 三、做分析:看是"算"还是"想"

数据分析要分两种情况:

### 1. 需要严谨推理的分析

比如解读复杂报表、做多维度归因、写分析报告——这种需要模型"想得深",**推荐 Opus**,它在长链条推理上更稳。

### 2. 简单的提取与汇总

比如从一段文本里抽关键信息、做简单分类、生成摘要——这种 **Sonnet 就够**,没必要用顶配,既快又省。

> 提醒一句:涉及具体数值计算时,无论哪个模型都可能出错,关键数字一定要自己复核,或者让它写代码来算,而不是直接口算。

## 四、一张总表

| 场景 | 子类型 | 推荐 |
|---|---|---|
| 写代码 | 日常 / 小改动 | Sonnet |
| 写代码 | 复杂 / 重构 | Opus |
| 写文案 | 日常改写 | Sonnet |
| 写文案 | 长篇 / 高要求 | Opus |
| 做分析 | 深度推理 | Opus |
| 做分析 | 提取汇总 | Sonnet |

## 五、通用原则

不管什么场景,我的选择逻辑都一样:

1. **先用便宜的试**(Sonnet 这类),大概率就成了。
2. **不满意再升级**到更强的(Opus)。
3. **重要产出直接上强的**,这点钱省不得。

把"默认便宜、按需升级"养成习惯,长期下来省的不是一点半点,而效果几乎不打折。

## 小结

没有"最好的模型",只有"最合适的模型"。写代码看复杂度、写文案够用就好、做分析看是算还是想。把任务和模型匹配对,才能既拿到好结果,又不花冤枉钱。
